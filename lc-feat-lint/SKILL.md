# 代码质量检查

分析功能代码的质量问题，**执行 ESLint 检查并自动修复可修复问题**，对安全的代码问题也自动修复，输出报告。

---

**Input**: `/lc-feat:lint <feat-name>`

**Steps**

1. **确定检查范围**

   读取 `docs/pipeline/{feat-name}/design.md` 或 `requirement.md` 获取本次功能涉及的文件列表。
   如果没有文档，通过 `git diff --name-only master` 获取变更文件列表。

2. **执行 ESLint 检查与自动修复**

   ### 2.1 首次检查

   ```bash
   npx eslint {file-list} 2>&1
   ```

   ### 2.2 区分新增代码 vs 原有代码的错误

   对比变更文件在 master 分支上是否已有相同的 ESLint 错误：
   - **原有错误**（master 上也存在的）：记录但不阻塞
   - **新增错误**（本次改动引入的）：必须处理

   ### 2.3 ESLint 自动修复

   ```bash
   npx eslint {file-list} --fix 2>&1
   ```

   修复后重新检查，确认修复结果。

3. **安全自动修复（不影响逻辑）**

   对 ESLint `--fix` 无法修复的问题，以及阅读代码发现的问题，**仅修复以下不影响逻辑的类型**：

   ### 3.1 可自动修复（直接用 Edit 工具修改）

   | 类型 | 示例 | 修复方式 |
   |------|------|---------|
   | 未使用的 import | `import { foo } from 'bar'` 但 foo 未使用 | 删除该 import 行 |
   | 重复的 import | 同一模块 import 两次 | 合并为一行 |
   | import 排序 | 未按项目约定排序 | 调整顺序 |
   | 多余的空行/空格 | 连续多个空行 | 删除多余空行 |
   | 缺少分号/多余分号 | 风格不一致 | 按项目 prettier 配置统一 |
   | console.log 遗留 | 调试代码未删除 | 删除（仅限新增代码中的） |
   | 未使用的变量声明 | `const x = 1` 但 x 未使用 | 删除（仅限明确未使用且无副作用的） |
   | CSS 属性顺序 | stylelint 可自动修复的 | 调整顺序 |

   ### 3.2 禁止自动修复（只报告，不动代码）

   | 类型 | 原因 |
   |------|------|
   | 条件判断修改（`==` → `===`） | 可能改变逻辑行为 |
   | 变量重命名 | 可能影响其他文件引用 |
   | 类型标注修改 | 可能引入类型错误 |
   | 函数参数增删 | 影响调用方 |
   | 错误处理逻辑 | 影响异常流程 |
   | 任何涉及条件分支/数据流的改动 | 影响业务逻辑 |

   > **铁律：有任何不确定是否影响逻辑的，一律不自动修复，只在报告中标注。**

4. **代码分析（人工维度）**

   逐文件阅读改动代码，补充 ESLint 无法覆盖的维度：

   ### 4.1 项目规范检查（参考 `.claude/CLAUDE.md` 和项目规范技能）
   - 是否遵循项目的组件/请求/路由模式
   - 是否使用了正确的工具函数和组件库

   ### 4.2 逻辑完整性
   - 新增分支是否覆盖了所有场景
   - 条件判断是否遗漏
   - 错误处理是否到位

5. **生成报告**

   在 `docs/pipeline/{feat-name}/` 下生成 `lint-report.md`：

   ```markdown
   # Lint 检查报告

   ## 检查范围
   | 文件 | 类型 |
   |------|------|
   | {file} | 新建/修改 |

   ## ESLint 结果
   - 新增代码错误：{N} 个（ESLint --fix 修复 {M} 个）
   - 原有代码错误：{K} 个（不在本次改动范围）

   ## 安全自动修复（已执行）
   | 文件 | 问题 | 修复方式 |
   |------|------|---------|
   | {file:line} | 未使用的 import `xxx` | 已删除 |

   ## 需手动处理（涉及逻辑，不自动修复）
   | 文件 | 问题 | 建议 |
   |------|------|------|

   ## 项目规范检查
   | 文件 | 状态 | 说明 |
   |------|------|------|

   ## 结论
   {通过 / 有 N 个问题需手动处理}
   ```

6. **质量门控判定**

   ### 通过条件（自动继续）
   - 新增代码 ESLint 0 error（warning 可忽略）
   - 安全自动修复已全部完成
   - 项目规范检查无严重偏差

   ### 阻塞条件（暂停等待）
   - 新增代码有无法自动修复的 ESLint error（涉及逻辑）
   - 项目规范检查发现严重违规

   阻塞时输出：
   > "Lint 检查发现 {N} 个问题需手动修复（详见 lint-report.md）。修复后可重新执行 `/lc-feat:lint {feat-name}` 验证。"

**Guardrails**
- **ESLint 检查必须实际执行**（`npx eslint`），不能仅靠人工阅读
- ESLint `--fix` 可修复的直接修复，不需要确认
- **安全自动修复必须严格遵守 3.1 白名单**，白名单外的一律不动
- **绝对不修改任何涉及条件判断、数据流、函数签名的代码**
- 修复前后必须重新执行 ESLint 验证，确认没有引入新错误
- 原有代码的错误不计入门控判定
- 如果项目没有 ESLint 配置，跳过 ESLint 步骤，仅做人工分析
