# 代码验证（Lint + Test + QA）

对功能代码执行三阶段验证：静态检查 → 单元测试 → 综合质量分析，输出统一的 `verify-report.md`。

---

**Input**: `/lc-feat:verify <feat-name>`

**Steps**

1. **确定验证范围**

   读取 `docs/pipeline/{feat-name}/spec.md`（需求与设计文档；旧版功能回退 `design.md` 或 `requirement.md`）获取本次功能涉及的文件列表。
   如果没有文档，通过 `git diff --name-only master` 获取变更文件列表。

   并行读取已有产出：
   - `docs/pipeline/{feat-name}/spec.md`（旧版功能：`requirement.md` + `design.md`）
   - 所有功能相关的源代码文件

2. **加载项目编码规范**

   按 `~/.claude/skills/docs/lc-feat-coding-standard-loading.md` 流程加载项目编码规范，作为验证检查的技术标准。

---

## 阶段一：静态检查（Lint）

3. **执行 ESLint 检查与自动修复**

   ### 3.1 首次检查

   ```bash
   npx eslint {file-list} 2>&1
   ```

   ### 3.2 区分新增代码 vs 原有代码的错误

   对比变更文件在 master 分支上是否已有相同的 ESLint 错误：
   - **原有错误**（master 上也存在的）：记录但不阻塞
   - **新增错误**（本次改动引入的）：必须处理

   ### 3.3 ESLint 自动修复

   ```bash
   npx eslint {file-list} --fix 2>&1
   ```

   修复后重新检查，确认修复结果。

4. **安全自动修复（不影响逻辑）**

   对 ESLint `--fix` 无法修复的问题，**仅修复以下不影响逻辑的类型**：

   ### 可自动修复（直接用 Edit 工具修改）

   | 类型 | 示例 | 修复方式 |
   |------|------|---------|
   | 未使用的 import | `import { foo } from 'bar'` 但 foo 未使用 | 删除该 import 行 |
   | 重复的 import | 同一模块 import 两次 | 合并为一行 |
   | 多余的空行/空格 | 连续多个空行 | 删除多余空行 |
   | console.log 遗留 | 调试代码未删除 | 删除（仅限新增代码中的） |
   | 未使用的变量声明 | `const x = 1` 但 x 未使用 | 删除（仅限明确未使用且无副作用的） |

   ### 禁止自动修复（只报告，不动代码）

   | 类型 | 原因 |
   |------|------|
   | 条件判断修改（`==` → `===`） | 可能改变逻辑行为 |
   | 变量重命名 | 可能影响其他文件引用 |
   | 类型标注修改 | 可能引入类型错误 |
   | 函数参数增删 | 影响调用方 |
   | 任何涉及条件分支/数据流的改动 | 影响业务逻辑 |

   > **铁律：有任何不确定是否影响逻辑的，一律不自动修复，只在报告中标注。**

5. **代码分析（人工维度）**

   逐文件阅读改动代码，补充 ESLint 无法覆盖的维度：
   - 是否遵循项目的组件/请求/路由模式
   - 是否使用了正确的工具函数和组件库

## 阶段二：单元测试（Test）

6. **判断是否需要测试**

   ### 测试范围规则
   **只测新增的核心函数，不测页面组件本身。**

   核心函数：新增的**纯函数**（label 映射、数据格式化、枚举转换）、**工具函数**（计算、校验、数据变换）、**常量文件**（结构完整性）。

   不测：Vue 页面组件渲染/交互、Service 层 API 封装、路由配置、权限配置。

   **跳过条件**：如果本次改动没有新增符合条件的核心函数，直接跳过测试阶段，在报告中标注"无新增核心函数，测试跳过"。

7. **环境检测与测试执行**（有核心函数时）

   ### 7.1 检测测试框架
   ```bash
   npx vitest --version 2>&1
   ```
   如果 vitest 不可用且 jest 也不可用，跳过测试阶段并在报告中标注。

   ### 7.2 生成测试文件
   - 每个 it 只测一个行为
   - 断言具体值，不写 `toBeTruthy()` 之类的模糊断言
   - 覆盖正常值、边界值、异常值

   ### 7.3 执行测试
   ```bash
   npx vitest run {测试文件路径} --reporter=verbose 2>&1
   ```

   失败用例自动诊断修复，最多重试 3 轮。

   ### 7.4 回归测试
   如果项目有已有测试文件，运行全量测试确认无回归：
   ```bash
   npx vitest run --reporter=verbose 2>&1
   ```

## 阶段三：综合质量分析（QA）

8. **需求覆盖度验证**

   逐项对照需求文档，检查每个功能点是否已实现、数据字段是否完整、操作流程是否一致。

9. **代码质量六维度分析**

   ### 9.1 逻辑完整性
   - 条件分支是否有遗漏、异步操作是否有错误处理、数据加载是否有 loading 状态

   ### 9.2 边界情况
   - 表单校验、分页参数、删除/危险操作二次确认

   ### 9.3 安全性
   - XSS 风险（v-html）、权限控制、敏感数据脱敏、API 参数校验

   ### 9.4 性能
   - 重复请求、大列表分页/虚拟滚动、不必要的重渲染

   ### 9.5 用户体验
   - 操作反馈（toast/xAlert）、防重复点击、loading、错误信息友好性

   ### 9.6 变更影响分析（回归安全性）
   - `git diff master --name-only` 获取变更范围
   - Bug 模式检测（8 patterns）
   - 删除/重命名验证（grep 确认无残留引用）
   - 行为变更检测（条件分支、语义漂移、接口签名传播）

10. **修复严重问题**

    如果存在严重问题（影响功能/安全），使用 Edit tool 逐个修复，修复后更新报告。

---

## 生成报告

11. **输出统一验证报告**

    在 `docs/pipeline/{feat-name}/` 下生成 `verify-report.md`：

    ```markdown
    # {功能名称} 代码验证报告

    ## 验证日期: {YYYY-MM-DD}
    ## 项目类型: H5 移动端 / 后台管理

    ---

    ## 一、静态检查（Lint）

    ### ESLint 结果
    - 新增代码错误：{N} 个（--fix 修复 {M} 个）
    - 原有代码错误：{K} 个（不在本次改动范围）

    ### 安全自动修复（已执行）
    | 文件 | 问题 | 修复方式 |
    |------|------|---------|

    ### 需手动处理（涉及逻辑）
    | 文件 | 问题 | 建议 |
    |------|------|------|

    ### 项目规范检查
    | 文件 | 状态 | 说明 |
    |------|------|------|

    ---

    ## 二、单元测试（Test）

    ### 测试概览
    - 状态：{执行/跳过}
    - 新增用例: {N} 个，通过: {N}，失败: {N}
    - 回归测试: {通过/有问题/无已有测试}

    ### 测试用例清单
    | 文件 | 用例名称 | 状态 |
    |------|---------|------|

    ### 失败用例分析（未修复）
    | 用例 | 失败原因 | 建议 |
    |------|---------|------|

    ---

    ## 三、综合质量分析（QA）

    ### 需求覆盖度
    | 功能点 | 状态 | 备注 |
    |--------|------|------|
    覆盖率: {百分比}

    ### 问题清单
    #### 严重（必须修复）
    | # | 问题 | 文件:行号 | 类型 | 修复建议 |
    |---|------|----------|------|---------|

    #### 一般（建议修复）
    | # | 问题 | 文件:行号 | 类型 | 修复建议 |
    |---|------|----------|------|---------|

    ### 各维度评分
    | 维度 | 评分 | 说明 |
    |------|------|------|
    | 逻辑完整性 | /10 | |
    | 边界处理 | /10 | |
    | 安全性 | /10 | |
    | 性能 | /10 | |
    | 用户体验 | /10 | |
    | 变更影响 | /10 | |

    ### 变更影响分析
    - [ ] 无残留引用（grep 验证通过）
    - [ ] 行为变更已识别并确认合理
    - [ ] 无 Bug 模式命中

    ---

    ## 总评

    - QA 评分: {0-10}/10
    - Lint: {通过/有问题}
    - 测试: {通过/跳过/有失败}
    - 状态: {通过 / 有 N 个问题需处理}
    ```

12. **质量门控与输出**

    ### 通过条件
    - Lint：新增代码 0 error
    - Test：全部通过或合理跳过
    - QA：评分 >= 8，无严重问题

    ### 阻塞条件
    - 新增代码有无法自动修复的 ESLint error
    - 测试用例反复失败（3 轮修复后）
    - QA 发现严重问题未修复
    - QA 评分 < 6（需回到实现阶段）

    使用 **AskUserQuestion tool**：
    > "代码验证完成，QA 评分 {N}/10。Lint {状态}，测试 {状态}。{严重问题数} 个严重问题已修复。报告：`docs/pipeline/{feat-name}/verify-report.md`\n\n确认后执行 `/lc-feat:pr {feat-name}` 创建 PR。"

**Guardrails**
- **ESLint 检查必须实际执行**（`npx eslint`），不能仅靠人工阅读
- **安全自动修复严格遵守白名单**，白名单外一律不动
- **绝不修改任何涉及条件判断、数据流、函数签名的代码**
- 修复前后必须重新 ESLint 验证
- 原有代码错误不计入门控判定
- **测试只测新增核心函数**，不强行为 Service/Vue 组件写无意义测试
- 必须实际执行测试（`npx vitest run`），不能只写不跑
- 失败用例优先修复测试代码，不随意修改源代码，最多重试 3 轮
- QA 必须对照需求文档逐项检查，不凭印象
- **QA 检查标准必须与项目类型匹配**（H5 不检查后台权限体系，后台不检查移动端适配）
- 严重问题（影响功能/安全）必须修复
- QA 评分标准：8+ 可直接 PR，6-7 需修复后重新验证，<6 需回到实现阶段
