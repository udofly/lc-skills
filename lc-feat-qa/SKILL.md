# QA 分析

对功能进行全面质量分析，覆盖逻辑、边界、安全、性能等维度。

---

**Input**: `/lc-feat:qa <feat-name>`

**Steps**

1. **加载全部产物**

   并行读取：
   - `docs/pipeline/{feat-name}/requirement.md`
   - `docs/pipeline/{feat-name}/design.md`（如存在）
   - `docs/pipeline/{feat-name}/test-report.md`（如存在）
   - `docs/pipeline/{feat-name}/lint-report.md`（如存在）
   - 所有功能相关的源代码文件

2. **加载项目编码规范（最高优先级）**

   按以下顺序查找并加载项目编码规范，确定 QA 检查标准：

   1. 读取项目 `.claude/CLAUDE.md`（如存在）— 获取项目指定的编码规范技能名称
   2. 如果其中提到的编码规范技能在当前可用技能列表中存在，**加载该技能的完整规范作为 QA 标准**
   3. 如果以上均不存在，通过读取 `package.json` 和 `src/` 目录结构自行判断项目类型

3. **需求覆盖度验证**

   逐项对照需求文档，检查：
   - [ ] 每个功能点 / 改动点是否已实现
   - [ ] 页面清单 / 改动文件清单是否全部完成
   - [ ] 数据字段是否完整
   - [ ] 操作流程是否与需求一致

4. **代码质量分析**

   ### 4.1 逻辑完整性
   - 条件分支是否有遗漏
   - 异步操作是否有错误处理（try/catch）
   - 数据加载是否有 loading 状态
   - 列表/内容是否有空数据提示

   ### 4.2 边界情况
   - 表单校验是否覆盖所有必填和格式要求
   - 分页参数是否正确传递和重置
   - 删除/危险操作是否有二次确认

   ### 4.3 安全性
   - 是否有 XSS 风险（v-html 使用）
   - 权限控制是否到位（根据项目类型：后台检查按钮/页面级权限，H5 检查登录态）
   - 敏感数据是否脱敏展示
   - API 调用是否有参数校验

   ### 4.4 性能
   - 是否有不必要的重复请求
   - 大列表是否有分页/虚拟滚动
   - 组件是否有不必要的重渲染

   ### 4.5 用户体验
   - 操作成功/失败是否有提示（根据项目类型：toast/xAlert 或 MessagePlugin）
   - 表单提交是否有防重复点击
   - 加载过程是否有 loading 反馈
   - 错误信息是否友好

5. **生成 QA 报告**

   在 `docs/pipeline/{feat-name}/` 下生成 `qa-report.md`：

   ```markdown
   # {功能名称} QA 分析报告

   ## 分析日期: {YYYY-MM-DD}
   ## 项目类型: H5 移动端 / 后台管理

   ## 总评
   - QA 评分: {0-10}/10
   - 状态: {通过/有问题需修复}

   ## 需求覆盖度
   | 功能点 | 状态 | 备注 |
   |--------|------|------|

   覆盖率: {百分比}

   ## 问题清单
   ### 严重（必须修复）
   | # | 问题 | 文件:行号 | 类型 | 修复建议 |
   |---|------|----------|------|---------|

   ### 一般（建议修复）
   | # | 问题 | 文件:行号 | 类型 | 修复建议 |
   |---|------|----------|------|---------|

   ### 建议（可选优化）
   | # | 问题 | 文件:行号 | 类型 | 修复建议 |
   |---|------|----------|------|---------|

   ## 各维度评分
   | 维度 | 评分 | 说明 |
   |------|------|------|

   ## 结论
   {是否可以进入 PR 阶段}
   ```

6. **修复严重问题**

   如果存在严重问题：
   - 使用 Edit tool 逐个修复
   - 修复后更新 QA 报告中对应项状态

7. **输出确认**

   使用 **AskUserQuestion tool**：
   > "QA 分析完成，评分 {N}/10。{严重问题数} 个严重问题已修复。报告：`docs/pipeline/{feat-name}/qa-report.md`\n\n确认后执行 `/lc-feat:pr {feat-name}` 创建 PR。"

**Guardrails**
- QA 必须对照需求文档逐项检查，不能凭印象
- 严重问题（影响功能/安全）必须修复后才能进入 PR 阶段
- 不要过度挑剔代码风格问题（那是 lint 的职责）
- 性能问题只关注明显的问题，不做过度优化
- **QA 检查标准必须与项目类型匹配**（H5 不检查后台权限体系，后台不检查移动端适配）
- QA 评分标准：8+ 可直接 PR，6-7 需修复后重新 QA，<6 需回到实现阶段
