# 需求分析

将用户输入（文字/截图/文档）转化为结构化需求文档，存储到 `docs/pipeline/{feat-name}/requirement.md`。

---

**Input**: 用户的需求描述，可以是文字、截图路径、或外部文档链接。参数格式：`/lc-feat:requirement <feat-name> [描述]`

**Steps**

1. **获取需求信息**

   如果用户没有提供足够的需求描述，使用 **AskUserQuestion tool** 询问：
   > "请描述你想要实现的功能，可以提供文字说明、截图路径或相关文档。"

   从描述中提取功能名称（kebab-case），如 "添加商品管理" → `goods-manage`。

2. **识别需求类型**

   根据用户描述判断需求类型，不同类型决定后续分析重点和文档模板：

   | 类型 | 判断依据 | 分析重点 |
   |------|---------|---------|
   | **新功能** | 需要新页面、新路由 | 页面清单、数据字段、完整流程 |
   | **功能扩展** | 在现有模块上增加分支/选项 | 影响面分析、改动文件清单 |
   | **Bug 修复** | 修复现有功能的异常 | 问题复现、根因定位、修复方案 |
   | **重构优化** | 不改变功能，改善代码/性能 | 改动范围、风险评估 |

3. **分析需求上下文**

   - 阅读用户提供的截图（如有，使用 Read tool 查看图片）
   - 分析用户提供的文档内容
   - **检测当前项目类型**（通过读取 `package.json`、`src/` 目录结构判断）：
     - H5 移动端项目（如 kejinshou_m）：Vant + Vue 3 + 自定义组件
     - 后台管理项目（如 backend-kejinshou）：KR36 + Vue 3 + CRUD 模板
   - 了解当前项目已有的相关模块（根据项目类型检查对应目录）
   - 检查是否有相关的 Apifox 接口文档可用（通过 MCP 工具，工具名从当前可用的 apifox MCP 工具列表中选取）

4. **影响面分析**（功能扩展/Bug修复/重构 类型必须执行）

   这是最关键的步骤，防止遗漏需要同步改动的文件：

   - **找到改动入口**：确定需要修改的核心代码点（如枚举、配置、组件）
   - **搜索所有引用**：使用 Grep 工具搜索核心代码点的所有引用位置
   - **逐个判断是否需要同步改动**：对每个引用点，分析其逻辑是否依赖于本次改动
   - **输出影响面清单**：列出所有需要改动的文件，标注改动类型和原因

   > 示例：扩展支付方式时，不仅要改 `PayMethodEnum` 枚举和 `PayManager.ts` 分发逻辑，
   > 还要 grep `PayMethodEnum.pay_fh_t_qr` 等类似枚举的引用，发现 `PopupPay.vue` 和
   > `PaymentMethod.vue` 中有支付方式列表构建逻辑也需要同步添加。

5. **生成需求文档**

   创建目录 `docs/pipeline/{feat-name}/`，根据需求类型选择对应模板生成 `requirement.md`。

   ### 模板 A：新功能（需要新页面/新路由）

   ```markdown
   # {功能名称} 需求文档

   ## 基本信息
   - **功能名称**: {中文名}
   - **功能标识**: {feat-name}
   - **需求类型**: 新功能
   - **创建日期**: {YYYY-MM-DD}
   - **优先级**: P0/P1/P2
   - **状态**: 待设计

   ## 需求背景
   {为什么要做这个功能，解决什么问题}

   ## 功能描述
   ### 核心功能
   - {功能点1}
   - {功能点2}

   ### 页面清单
   | 页面 | 路径 | 说明 |
   |------|------|------|
   | {页面名} | /xxx/xxx | {说明} |

   ### 用户操作流程
   1. {步骤1}
   2. {步骤2}

   ## 数据需求
   ### 列表字段
   | 字段名 | 类型 | 说明 |
   |--------|------|------|
   | {字段} | {类型} | {说明} |

   ### 搜索/筛选条件
   | 条件 | 类型 | 说明 |
   |------|------|------|
   | {条件} | {类型} | {说明} |

   ### 表单字段（新增/编辑）
   | 字段名 | 类型 | 必填 | 校验规则 | 说明 |
   |--------|------|------|----------|------|
   | {字段} | {类型} | 是/否 | {规则} | {说明} |

   ## 接口需求
   {从 Apifox 读取的接口信息，或标注"待定 - 使用 mock 数据"}

   ## 非功能需求
   - 性能要求: {如有}
   - 兼容性: {如有}

   ## 参考资料
   - {截图/文档/原型链接}
   ```

   ### 模板 B：功能扩展 / Bug修复 / 重构优化

   ```markdown
   # {功能名称} 需求文档

   ## 基本信息
   - **功能名称**: {中文名}
   - **功能标识**: {feat-name}
   - **需求类型**: 功能扩展 / Bug修复 / 重构优化
   - **创建日期**: {YYYY-MM-DD}
   - **优先级**: P0/P1/P2
   - **状态**: 待设计

   ## 需求背景
   {为什么要做这个改动，解决什么问题}

   ## 功能描述
   ### 改动内容
   - {改动点1}
   - {改动点2}

   ### 用户操作流程（改动后）
   1. {步骤1}
   2. {步骤2}

   ## 影响面分析
   ### 改动文件清单
   | 文件 | 改动类型 | 改动说明 | 原因 |
   |------|---------|---------|------|
   | {文件路径} | 新增/修改/删除 | {具体改动} | {为什么这个文件需要改} |

   ### 改动范围评估
   - **改动量**: 极小/小/中/大
   - **风险等级**: 低/中/高
   - **核心改动**: {最关键的 N 个文件}
   - **联动改动**: {因核心改动需要同步修改的文件}

   ## 接口需求
   {是否需要新增/修改接口，或"无接口变更"}

   ## 非功能需求
   - 兼容性: {如有}

   ## 参考资料
   - {截图/文档/原型链接}
   - {相关代码位置引用，格式 file_path:line_number}
   ```

6. **输出确认**

   展示生成的需求文档摘要，使用 **AskUserQuestion tool** 确认：
   > "需求文档已生成，请查看 `docs/pipeline/{feat-name}/requirement.md`。是否需要修改？确认后可执行 `/lc-feat:design {feat-name}` 进入设计阶段。"

   对于改动量极小的功能扩展/Bug修复，额外提示：
   > "本次改动量较小，可跳过设计阶段直接编码。确认后执行 `/lc-feat:implement {feat-name}` 或直接手动实现。"

**Guardrails**
- 如果用户提供了截图，必须使用 Read tool 读取并分析截图内容
- 尽量从 Apifox MCP 获取接口信息，获取不到时标注"待定 - 使用 mock 数据"
- **功能扩展型需求必须执行影响面分析**，grep 所有相关引用，不能只看核心入口文件
- 需求文档中的"改动文件清单"必须逐文件标注改动原因，不能写"无需改动"然后遗漏
- 根据项目类型使用对应术语（H5 项目不要写 KrCard/KrTable 等后台组件名）
- 需求类型不同，后续流程建议也不同：小改动可跳过 design/test/qa 直接实现
