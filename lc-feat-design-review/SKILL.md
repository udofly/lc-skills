# 设计自审 & 架构评审

根据用户提供的参考材料（截图/文档/文字），自审设计文档并进行架构层面的深度评审，修订完善。

---

**Input**: `/lc-feat:design-review <feat-name> [参考材料说明]`

用户可能提供：
- 截图路径（UI 原型、竞品截图、现有页面截图）
- 文字补充说明
- 外部文档链接或内容

**Steps**

1. **加载设计文档和需求文档**

   并行读取：
   - `docs/pipeline/{feat-name}/design.md`
   - `docs/pipeline/{feat-name}/requirement.md`

   如果设计文档不存在，提示先执行 `/lc-feat:design`。

2. **加载项目编码规范（最高优先级）**

   按以下顺序查找并加载项目编码规范，确定审查使用的技术标准：

   1. 读取项目 `.claude/CLAUDE.md`（如存在）— 获取项目指定的编码规范技能名称
   2. 如果其中提到的编码规范技能在当前可用技能列表中存在，**加载该技能的完整规范作为审查标准**
   3. 如果以上均不存在，通过读取 `package.json` 和 `src/` 目录结构自行判断项目类型

3. **收集参考材料**

   如果用户没有提供参考材料，使用 **AskUserQuestion tool** 询问：
   > "请提供参考材料用于审核设计文档，可以是：\n1. 截图路径（UI 原型/竞品页面）\n2. 文字补充说明\n3. 或直接说'基于需求文档自审'跳过"

   - 如果提供了截图路径，使用 **Read tool** 读取并分析截图
   - 如果提供了文字说明，整合到审核依据中

4. **执行设计审查**

   按以下维度逐项检查：

   ### A. 需求覆盖度检查
   - [ ] 需求文档中的每个功能点是否都有对应设计
   - [ ] 页面清单 / 改动文件清单是否完整覆盖
   - [ ] 数据字段是否全部涵盖
   - [ ] 影响面分析是否完整（功能扩展型需求特别重要）

   ### B. UI/交互一致性检查（对照截图）
   - [ ] 页面布局是否与参考截图一致
   - [ ] 交互流程是否匹配
   - [ ] 操作按钮/入口是否完整

   ### C. 技术合理性检查（根据项目类型）

   **H5 移动端项目**：
   - [ ] 是否使用了正确的组件（Vant / XPopup / XNavBar 等）
   - [ ] Request 层是否遵循 `src/request/MwpXxx.ts` 模式
   - [ ] 路由是否使用 `routerName` + `routerPathKey` 模式
   - [ ] 提示方式是否使用 `toast()` / `xAlert()`
   - [ ] 事件通信是否使用 `emitter` 模式

   **后台管理项目**：
   - [ ] 组件选型是否合适（KrCard/KrForm/KrTable/KrDialog）
   - [ ] Service 层是否遵循 `ApiUrl` + `request.post/get` 模式
   - [ ] 路由是否使用 LAYOUT 常量 + `meta.permissions`
   - [ ] 提示方式是否使用 `MessagePlugin`

   **通用**：
   - [ ] 是否存在遗漏的边界场景（空数据、加载状态、错误处理）
   - [ ] 是否与项目现有相似模块的模式保持一致

   ### D. 架构评审（深度）

   #### D.1 数据流分析
   - [ ] 组件间数据流向是否清晰（props down / events up / emitter）
   - [ ] 是否存在不必要的状态提升或跨组件通信
   - [ ] 是否需要 Pinia store，还是组件内 ref 足够

   #### D.2 组件边界
   - [ ] 组件职责是否单一
   - [ ] 子组件拆分粒度是否合适
   - [ ] 与项目中已有组件是否有重复

   #### D.3 性能瓶颈预判
   - [ ] 列表数据量预估，是否需要虚拟滚动/懒加载
   - [ ] 搜索/输入是否需要防抖
   - [ ] 接口调用频率是否合理

   #### D.4 扩展性评估
   - [ ] 设计是否便于后续迭代
   - [ ] 接口参数设计是否预留了扩展空间

   #### D.5 与现有架构一致性
   - [ ] 是否与项目中同类模块的架构保持一致
   - [ ] 是否引入了新的模式或依赖（需有充分理由）

   ### E. 项目规范一致性检查
   - [ ] 文件命名是否遵循项目规范
   - [ ] 代码风格是否与项目一致（Composition API + script setup）
   - [ ] 与现有模块是否存在重复或冲突

5. **生成审查报告并修订**

   在 `docs/pipeline/{feat-name}/` 下生成 `design-review.md`：

   ```markdown
   # {功能名称} 设计审查报告

   ## 审查日期: {YYYY-MM-DD}
   ## 项目类型: H5 移动端 / 后台管理
   ## 参考材料: {截图/文档/文字说明概述}

   ## 审查结果总览
   - 需求覆盖度: {百分比}
   - UI 一致性: {通过/存在偏差}
   - 技术合理性: {通过/需修改}
   - 架构评审: {通过/需修改}
   - 规范一致性: {通过/需修改}

   ## 发现的问题
   ### 严重问题（必须修复）
   1. {问题描述} → {修复方案}

   ### 建议优化
   1. {建议描述} → {优化方案}

   ## 修订记录
   | 项目 | 原设计 | 修订后 | 原因 |
   |------|--------|--------|------|
   | {项} | {原} | {新} | {原因} |
   ```

6. **自动修订设计文档**

   根据审查发现的问题，直接修改 `design.md`：
   - 使用 **Edit tool** 逐项修订
   - 在设计文档头部更新状态为 "已审核-修订"
   - 添加修订版本记录

7. **输出确认**

   展示审查结果摘要和修订内容，使用 **AskUserQuestion tool**：
   > "设计审查完成，共发现 {N} 个问题已修订。请查看：\n- 审查报告：`docs/pipeline/{feat-name}/design-review.md`\n- 修订后设计：`docs/pipeline/{feat-name}/design.md`\n\n确认设计无误后，执行 `/lc-feat:implement {feat-name}` 开始编码。"

8. **质量门控判定**

   根据审查结果分级：

   ### 通过条件（自动继续到 implement）
   - 无严重问题，或严重问题已全部修订
   - 需求覆盖度 >= 90%

   ### 阻塞条件（暂停等待确认）
   - 有 P1 级严重问题（如架构方向错误、核心组件选型不当、数据流设计缺陷）
   - 需求覆盖度 < 80%（有功能点未在设计中体现）

   阻塞时使用 **AskUserQuestion** 提示：
   > "设计审核发现 {N} 个严重问题（已自动修订）。请确认修订后的设计文档是否满意，或需要进一步调整后再进入编码阶段。"

**Guardrails**
- 必须对照需求文档逐项检查，不能跳过
- 如果用户提供了截图，必须逐个 UI 元素对比
- **技术合理性检查必须基于项目类型**，H5 项目不能用后台标准审查，反之亦然
- 发现问题必须同时给出修复方案并执行修订
- 修订设计文档后，原有结构不能被破坏
- 审查报告必须保留，作为设计决策记录
- P1 级问题必须阻塞流程，不能静默通过
