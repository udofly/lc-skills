# 文档时间同步（lc-doc-sync）

按**时间/commit 粒度**兜底同步项目文档。`lc-feat:document-release` 在 PR 合并后按**功能粒度**精细更新本功能的文档，但 hotfix、重构、直接提交、他人合并的功能会绕过流水线——本技能负责把这些"漏网"变更对应的文档补齐。两者共写同一本更新账本、同一套 frontmatter 时间戳，互为补充而非替代。

---

**Input**:

| 命令 | 模式 |
|------|------|
| `/lc-doc-sync` | 增量：账本 `last_sync.commit` → HEAD |
| `/lc-doc-sync --since=<日期或commit>` | 从指定起点 → HEAD（日期用 `git rev-list -1 --before=<日期> HEAD` 折算成 commit） |
| `/lc-doc-sync --scan` | 全局扫描：逐篇比对文档与源码新鲜度 |
| 任意模式 + `--check` | 只输出报告，**不修改任何文件** |

**Steps**

1. **读取项目文档约定（禁止硬编码任何项目的路径/结构）**

   本 skill 通用于所有项目，所有路径、结构、模板均在运行时读取：

   1. 读项目 `.claude/CLAUDE.md`，找到它引用的**项目编码规范技能**（如 `lc-kejinshou-h5-vue`）。
   2. 读该技能的**「文档维护约定」**章节，获取：文档根目录、业务文档结构（如 1:1 镜像 `src/views`）、页面模板、命名规则、侧边栏文件、校验命令，以及**更新账本位置、记录行格式、frontmatter 时间戳与幂等约定**。
   3. **回退**：项目未声明「更新账本」→ 说明需要先建账本，经用户确认后代为初始化（在文档根目录创建 `updates.md`，锚点取用户指定的 commit 或当前 HEAD），并提醒把账本约定补进项目技能的「文档维护约定」。

2. **确定同步范围**

   - 增量模式：锚点 = 账本 frontmatter `last_sync.commit`。先 `git cat-file -t <锚点>` 确认存在；不存在（浅克隆/换库）→ 提示用 `--since` 指定。
   - `--since` 模式：忽略账本锚点，用用户给定的起点。
   - `--scan` 模式：不需要区间，跳到 Step 5 的扫描流程。

3. **收集变更并映射到文档**

   ```bash
   git diff --name-status <起点>..HEAD -- src/   # 变更文件清单（源码目录以项目约定为准）
   git log --oneline <起点>..HEAD -- <file>      # 需要理解某文件"改了什么"时再查
   ```

   按项目约定把源码文件映射到文档（以下是 1:1 镜像结构的通用映射，具体以项目「文档维护约定」为准）：

   | 变更源码 | 目标文档 |
   |----------|----------|
   | 页面 `.vue`（业务文档目录所镜像的源码目录） | 对应页面文档；**新增页面或文档缺失**（历史漏建的旧页面同样处理）→ 按模板新建 + 更新模块 index + 挂侧边栏 |
   | 页面子组件（`components/` 子目录） | grep 找到引用它的页面 → 该页面文档「组件依赖」段；引用方是**公共组件**而非页面 → 归入公共组件顶层文档 |
   | 路由文件 | diff 出受影响的路由 → 对应页面文档「路由」段 |
   | 请求层文件 | grep 找到引用页面 → 页面文档「接口依赖」表；项目有顶层 api 文档也一并更新 |
   | 公共组件 / 工具 / composables | 项目声明的对应顶层文档（components / utils 等） |
   | 删除的页面 | 对应文档标注废弃或删除（删除需用户确认），并从模块 index / 侧边栏摘除 |

   **映射不到的变更**（如构建配置、类型定义）列入报告的「未映射变更」清单——宁可让用户看到，不可静默丢弃。

   **同步范围只含已提交内容**：`git diff <起点>..HEAD` 覆盖不到工作区未提交改动。`git status` 发现源码目录有未提交改动时，在报告附注中列出并注明"提交后纳入下次同步"，避免用户误以为已覆盖。

4. **幂等过滤**

   对每篇目标文档读 frontmatter：`commit` 等于变更 commit 或是其后代（`git merge-base --is-ancestor <变更commit> <文档commit>` 为真）→ 该文档已被更新过（通常是 `lc-feat:document-release` 干的），跳过。这保证两个技能重叠运行不会重复劳动。

   **文档无 frontmatter 戳时**（常见于建档基线未打戳）：回退为比较文档文件与源码文件各自的最后提交时间（`git log -1 --format=%ct -- <路径>`），文档不早于源码 → 跳过。与 `--scan` 的新鲜度回退是同一规则。

5. **更新文档（`--scan` 从这里开始）**

   - `--scan` 判定新鲜度：对每篇页面文档反推源码路径，比较文档 frontmatter `commit` 与 `git log -1 --format=%H -- <源码文件>`；文档无 frontmatter 时退化为比较两者各自的最后提交时间。源码更新 → 过时。
   - **`--check`：输出报告后立即结束**（全量过时清单、每篇的落后区间、未映射变更、未提交改动附注），不做任何修改，也不触发圈范围提问。
   - 实际更新时，过时文档 **> 15 篇** → 先展示清单并用 AskUserQuestion 让用户选择范围（全部 / 只核心模块 / 自选），避免一次生成海量低质更新。
   - 更新方式遵循项目「更新规则」：Edit 相关字段（接口/交互/跳转/坑），不重写整篇；内容**以代码为准**、只写业务层面，禁止臆测。

6. **打戳、记账、推进锚点**

   - 每篇更新过的文档 frontmatter 写 `updated: YYYY-MM-DD` + `commit: <本次同步到的 HEAD 短 hash>`。
   - 账本「更新记录」表最上方追加一行：日期 / 同步范围 / 来源（`lc-doc-sync 增量` / `--since` / `--scan`）/ 更新文档清单 / 摘要。
   - **完整同步到 HEAD 时**（增量或 `--since` 全量执行完、或 `--scan` 全部更新完）推进账本 `last_sync` 为 `{commit: HEAD短hash, date: 今天, by: lc-doc-sync}`；用户只选了部分文档更新 → **不推进锚点**，在记录摘要里注明"部分同步"。

7. **校验与报告**

   - 跑项目声明的校验命令（如 `pnpm docs:build`），失败先修复再交付。
   - 输出：更新了哪些文档（分新建/修改）、跳过的已同步文档、未映射变更、锚点是否推进；建议文档改动独立提交（`docs:` 前缀），**不自动 commit**。

---

**Guardrails**

- 所有路径/结构/模板来自项目技能「文档维护约定」，本 skill 不内置任何项目细节。
- `--check` 是硬承诺：任何文件都不改，包括账本。
- 锚点只增不减：新锚点必须是旧锚点的后代，否则说明用户在做回溯同步（`--since` 早于锚点），此时只记账不动锚点。
- 文档内容以代码为准，只写业务层面的"做什么/怎么用"，不写实现细节；改已有文档用 Edit 改相关字段，不重写整篇。
- 大批量更新（>15 篇）必须先让用户圈定范围。
- 未映射变更必须出现在报告里，不静默丢弃。
- 不自动 git commit；建议独立的 `docs:` 提交。
