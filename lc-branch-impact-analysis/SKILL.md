# 分支影响点分析与业务安全性检查 (Branch Impact Analysis)

本 Skill 用于自动化分析当前 Git 分支与目标分支（通常为 `master`）的代码差异，提取影响点，并着重评估新增逻辑是否做到了环境隔离，确保不影响原有核心业务。

适用于 Vue/Nuxt 前端项目、Android 项目等。

## 执行步骤

### 1. 获取代码差异与文件分类

```bash
git diff master --name-status
```

**按优先级分类处理**（大量文件时避免遗漏核心变更）：

| 优先级 | 文件类型 | 处理方式 |
|---|---|---|
| P0 | 核心工具/服务/公共模块（`utils/`、`composables/`、`plugins/`、`stores/`、`core/`、`services/`） | 逐文件 diff 分析 |
| P1 | 页面/Activity/配置（`pages/`、`views/`、`activity/`、构建配置） | 逐文件 diff 分析 |
| P2 | 组件/Fragment（`components/`、`widgets/`）| 关注 Props/接口/逻辑变更，纯重命名(R100)可跳过 |
| P3 | 样式/资源/测试/文档 | 概要说明，不逐行分析 |

- 对 R100（100% 重命名）文件仅记录路径变化，不分析内容
- 对 D（删除）文件重点检查是否有残留引用
- 聚焦 M（修改）和 A（新增）文件

### 2. 分析与分类影响点

将变更归类，例如：
- "单元测试变更"
- "新增业务逻辑"
- "全局/公共配置变更"
- "架构重构（如依赖注入方式变更、状态管理迁移）"
- "组件/模块迁移/重命名"
- "API 层变更"
- "权限/安全相关变更"

分析每类变更的具体业务含义。

### 3. 删除资产验证（主动 grep）

**对所有被删除的公共函数/组件/类/模块，必须 grep 全局确认无残留引用：**

```bash
grep -rn '符号名' --include='*.vue' --include='*.ts' --include='*.js' --include='*.kt' --include='*.java' --include='*.xml' | grep -v node_modules | grep -v .nuxt | grep -v dist | grep -v build/
```

- grep 结果不为空 → 🔴 残留引用 Bug
- 对重命名的组件/类，确认新旧名称映射正确
- 对 import/依赖路径变更，确认新路径文件存在

### 4. 语义等价性验证

对重构的函数/方法（名字变了但逻辑应等价）：
- **对比新旧实现**，列出关键输入场景，确认输出一致
- 对函数签名变更（如参数增减、类型变更），确认调用方已同步修改
- 对判断逻辑变更，列出行为差异的具体输入值

示例思路：
> 旧函数用 `includes('h5-')` 匹配所有以 h5- 开头的渠道；新函数用 `startsWith('h5-nature')` 仅匹配自然流量渠道。`h5-ali`、`h5-mp-baidu` 等不再匹配 → **这不是等价重构，是语义变更。**

### 5. 配置与路由/导航分析

当变更涉及以下类型文件时，需单独分析其全局影响：

**前端项目：**

| 文件 | 关注点 |
|---|---|
| 构建配置（nuxt/vite/webpack config） | viewport、experimental、modules、代理配置 |
| 路由配置 | 自动路由 + 手动路由并存是否冲突、路由被删除是否影响导航 |
| 包管理（package.json） | 依赖版本变更、scripts 变更 |
| 环境变量（.env.*） | 运行时变量变更 |
| 样式配置（tailwind/postcss config） | 全局样式影响 |

**Android 项目：**

| 文件 | 关注点 |
|---|---|
| build.gradle / build.gradle.kts | 依赖版本、编译选项、签名配置 |
| AndroidManifest.xml | 权限、Activity 声明、intent-filter |
| proguard-rules.pro | 混淆规则变更可能导致运行时崩溃 |
| Navigation graph | 导航路径变更 |

### 6. 安全性与环境隔离评估（核心）

- 检查新增的业务逻辑是否使用了严格的条件判断进行安全隔离
  - 前端：平台检测（`isPC`、`isMiniApp`、`isApp` 等）
  - Android：版本判断（`Build.VERSION.SDK_INT`）、渠道判断、Feature Flag
- 验证原有业务逻辑的执行分支是否保持完整
- 确保遇到非新增场景时，依然执行原有逻辑（如 `if (newCondition) { ... return; }` 之后原有逻辑未被篡改）

### 7. 跨文件影响链追踪

对架构级重构，追踪完整依赖链路：

```
提供方（plugin/module/provider） → 中间层（composable/工具类/service） → 消费方（页面/组件/Activity）
```

验证链路中每一环都已正确迁移，无断裂。

### 8. 临时代码与 TODO 检测

扫描新增代码中的临时标记：
- `// todo`、`// hack`、`// fixme`、`// temporary`
- 未说明原因的 `return;` 跳过逻辑
- 被注释掉的整块代码
- 临时 URL、测试地址、硬编码凭证

标记为 🟡 提醒用户处理。

## 输出报告（中文）

```markdown
## 🌿 分支影响点分析报告

### 📊 变更范围总览

| 类别 | 数量 | 说明 |
|---|---|---|
| 修改文件 (M) | X | 逻辑修改 |
| 新增文件 (A) | X | 新组件、测试、配置 |
| 删除文件 (D) | X | 旧组件、样式、资源 |
| 重命名文件 (R) | X | 路径迁移 |

### 逐项分析

#### 1. [变更类别名] — [🔴/🟡/🟢 风险等级]

**改动**：具体描述
**评估**：隔离方式 / 等价性验证结果 / grep 验证结果
**结论**：对原有业务的影响

#### 2. ...

### 🔴 Bug（需修复）
- `文件:行号` — 具体问题描述（附 grep 验证结果）

### 🟡 行为变更（需确认）
- `文件:行号` — 变更描述，新旧行为对比

### ⚙️ 配置变更
- `配置文件` — 变更内容及全局影响

### ⚠️ 临时代码
- `文件:行号` — todo/hack 描述

### 📋 风险汇总

| 风险项 | 等级 | 文件 | 说明 |
|---|---|---|---|
| 具体风险 | 🔴/🟡/🟢 | 文件:行号 | 简要说明 |

### ✅ 最终结论
- 对原有业务的整体影响评估
- 需要重点验证的项目
```

报告应结构化、每条问题包含文件名和行号。风险汇总表始终输出。无 Bug 或行为变更时，对应区块可省略。
